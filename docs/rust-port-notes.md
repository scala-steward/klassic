# Rust Port Notes

This file records intentional Rust-port decisions that are not obvious from the
code alone. The detailed feature inventory lives in `docs/spec-matrix.md`, and
the module layout is described in `docs/architecture-rust.md`.

## Scope Decisions

- The default build and runtime path is Rust-native through Cargo.
- Java/JVM interop surfaces from the historical implementation are intentionally
  out of scope for the native Rust port.
- The evaluator remains the semantic reference for required non-Java-FFI
  language behavior.
- The x86_64 native compiler is intentionally marked partial until it reaches
  evaluator parity across every required feature. Unsupported native constructs
  fail at build time rather than silently falling back to the evaluator.

## Typechecker Decisions

- Type variables and row variables are no longer treated as dynamic escapes
  during unification. They are resolved and propagated into annotations, so
  polymorphic result mismatches such as binding `id("oops")` to `Int` are
  rejected.
- Function-call arguments provide contextual types to inline lambdas. This keeps
  curried reducers such as `foldLeft(xs)([])((acc, e) => e #cons acc)` precise
  while preserving the lightweight Klassic syntax.
- Nominal record values can satisfy structural row field functions. A function
  like `def get_x(o) = o.x` remains row-polymorphic across record shapes, while
  later numeric use of that field fixes and checks the field type.

## Native Compiler Decisions

- Native codegen emits Linux x86_64 ELF64 directly from Rust without invoking an
  assembler, linker, JVM, Scala, or sbt.
- Static strings, lists, maps, sets, records, lambdas, builtin function values,
  and virtual File/Dir state are tracked by the compiler so the current native
  subset can preserve observable side effects while still folding known values.
- Function and builtin-function equality follows evaluator behavior for user
  code: equality returns false even for the same function value. Branch-state
  merging uses a separate structural comparison so equivalent closures can still
  survive native dynamic joins.
- Native runtime failures that are generated by the executable itself should
  include source-location prefixes on stderr. This currently covers `ToDo`,
  `assert`, failing `assertResult`, empty `head`, negative `sleep`, negative
  string helper indexes/counts, FileOutput syscall failures, and Dir
  mkdir/delete/move syscall failures.
- `Dir#mkdirs` intentionally ignores `EEXIST`, matching `create_dir_all`.
  `FileOutput#delete` intentionally ignores `ENOENT`, matching the evaluator's
  successful delete of missing files.
- `Dir#copy` uses static virtual-file content when the source was produced by
  earlier native FileOutput code. Otherwise it emits runtime `open`/`sendfile`
  code, marks the target path unknown for later static File/Dir folds, and lets
  the executable report source-located failures.
- `Dir#move` preserves known virtual file/dir facts when the source is virtual.
  For external runtime paths it emits `rename` and marks the source and target
  unknown so later folds cannot rely on stale build-time filesystem state.
- `Dir#current()` emits a runtime `getcwd` syscall and returns a fixed-buffer
  runtime string, so generated native executables observe their execution cwd
  instead of the cwd used during native build.
- `CommandLine#args()` reads the generated executable's argv at runtime and
  returns a runtime line list of user arguments excluding argv[0]. Direct calls,
  unqualified calls, aliased helper values, and calls from generated native
  functions use the same saved process-argument state.
- `Process#exit(code)` evaluates its code argument, rejects negative native
  codes with a source-located runtime diagnostic, and otherwise emits the Linux
  `exit` syscall. This gives generated native CLI tools explicit status codes.
- `StandardInput#all()` / `stdin()` read stdin into fixed-buffer runtime
  strings, while `StandardInput#lines()` / `stdinLines()` expose stdin through
  the same runtime line-list representation used by file input and argv helpers.
- Direct native printing of `FileInput#all(path)` or `FileInput#readAll(path)`
  streams runtime file bytes into the selected output fd. This deliberately
  avoids manufacturing a heap string until the native runtime has general
  heap-backed string values.
- Immutable native bindings of runtime `FileInput#all(path)` / `readAll(path)`
  use a fixed runtime string buffer that can be printed or used in `+` string
  concatenation. They also support native byte-exact equality/inequality and
  `assertResult`, plus `isEmptyString` / `length`, method-style `toString`,
  `substring` / `at` with static or runtime integer indexes,
  ASCII-whitespace trimming, `repeat` with static or runtime integer counts,
  ASCII `toLowerCase` /
  `toUpperCase`, simple `matches` with static or runtime patterns,
  first-occurrence `replace` with static or runtime literal operands,
  all-occurrence `replaceAll`, UTF-8 `reverse`, `startsWith`, `endsWith`,
  method-style `contains`, and `indexOf` / `lastIndexOf`; `length`,
  `substring`, and `at`
  count UTF-8 code points for valid UTF-8 data, `reverse` preserves UTF-8 code
  point byte groups, while
  index helpers return byte offsets to match the existing static helper
  behavior. Files larger than 65536 bytes fail at runtime with a source-located
  diagnostic instead of being silently truncated.
- Native string interpolation still folds static fragments where possible, and
  now builds fixed-buffer runtime strings when fragments resolve to native
  runtime strings or dynamic native `Int` / `Boolean` values.
- Native `+` string concatenation also formats dynamic native `Int` /
  `Boolean` operands into fixed-buffer runtime strings when the other side is a
  static or runtime string.
- `FileOutput#write` / `FileOutput#append` can write fixed-buffer runtime
  string content. Because the resulting contents are not known at native build
  time, the affected path is treated as unknown for later compile-time file
  folding. Unknown-path `FileInput#all`, `FileOutput#exists`, `Dir#exists`,
  `Dir#isFile`, `Dir#isDirectory`, `Dir#list`, and `Dir#listFull` now emit
  runtime syscalls instead of rejecting the native build.
- Runtime string values can now be used as syscall paths for `FileInput#all`,
  direct `FileInput#all` / `readAll` printing, simple
  `FileInput#open(path, stream => FileInput#readAll(stream))` callbacks,
  direct printing or immutable printable bindings of `FileInput#lines` /
  `readLines`, and matching simple `open(...readLines...)` callbacks. Runtime
  line-list values also support `size`, `isEmpty`, `head`, `tail`, `cons`,
  inline or aliased-lambda `map` that produces string line lists,
  String-accumulator `foldLeft` with inline or aliased reducers, `join`,
  `split` / `join` with static or runtime string delimiters on runtime strings,
  runtime `foreach`, and
  equality / `assertResult` checks
  against static string lists or other runtime line lists, and `FileOutput#writeLines` can write
  runtime line-list values back out,
  `FileOutput#write` / `append` / `writeLines` / `exists` / `delete`,
  `Dir#mkdir` / `mkdirs` / `delete` / `copy` / `move`, and `Dir#exists` / `isFile` /
  `isDirectory` / `list` / `listFull`. Runtime native directory listings are
  represented as runtime line lists and sorted to match the evaluator-compatible
  static directory listing order. The native emitter copies path bytes into a
  bounded NUL-terminated buffer before each syscall.

## Known Native Gaps

- Runtime heap-backed strings and aggregate values are not yet general.
- Native FileInput runtime strings are currently fixed-buffer values; direct
  print streaming remains the path for arbitrary-size runtime file output.
- Runtime-string native `split` supports static or runtime string delimiters.
  Empty delimiters split UTF-8 code point byte groups into runtime line-list
  elements. Non-empty delimiter splitting preserves empty-input, leading,
  consecutive, and trailing empty fields to match the evaluator's Rust
  `str::split` behavior. Runtime line-list `join` likewise accepts static or
  runtime string delimiters.
- Unsupported native features remain documented as `partial` in the spec matrix
  and must produce build-time diagnostics instead of accidental wrong code.
